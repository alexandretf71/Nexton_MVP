## AI Implementation Copilot

**Version:** 1.0  
**Date:** 2026-06-02

---

## 1. Problem → Blueprint Pipeline

The core workflow is a single-call LLM pipeline:

```
Business Problem Text
        │
        ▼
[Prompt Builder]
  system_prompt = role framing + output schema instruction
  user_prompt   = problem_description + optional context
        │
        ▼
[LLM Provider]
  Mock  → returns deterministic JSON fixture
  Real  → Anthropic Claude API, JSON mode
        │
        ▼
[Response Parser]
  json.loads(raw) → BlueprintOutput.model_validate(dict)
        │
        ▼
Validated BlueprintOutput (10 sections)
```

---

## 2. LLM Call Strategy: Single-Shot for MVP

**Decision:** One LLM call produces all 10 output sections simultaneously.

**Pros:**
- Lowest latency (one round-trip)
- Simplest error handling
- Easiest to test deterministically

**Cons:**
- Output quality is bounded by what a single call can reliably produce
- A failure in one section fails the whole response
- No section-specific refinement

**v2 Enhancement:** Multi-step agentic pipeline where each section is generated by a specialized sub-agent with its own prompt and validation. This produces higher quality but requires orchestration, retry logic, and streaming.

## 3. System Prompt Design

The system prompt lives in `app/core/prompt_templates.py` as a module-level constant (enabling Anthropic prompt caching).

### Objectives of the System Prompt
1. **Role framing** — Establish the LLM as a senior AI solutions architect with business translation skills.
2. **Output format enforcement** — Require strict JSON conforming to the `BlueprintOutput` schema.
3. **Tone calibration** — Professional, direct, CTO-appropriate language.
4. **Scope boundary** — Focus on the business problem, not general AI advice.

### System Prompt Structure

```
You are a senior AI solutions architect and implementation lead with deep experience 
translating messy operational problems into actionable AI implementation plans.

Your task is to analyze the business problem provided and return a complete AI 
implementation blueprint as a single, valid JSON object.

CRITICAL REQUIREMENTS:
- Return ONLY valid JSON. No prose, no markdown, no code fences.
- Every required field must be populated. Do not omit sections.
- Use professional, concise language appropriate for a CTO audience.
- Base your recommendations on the specific problem described, not generic AI advice.

OUTPUT SCHEMA:
[Full JSON schema of BlueprintOutput injected here]

FIELD GUIDANCE:
- business_problem_diagnosis: Restate and clarify the problem. Identify root causes. 
  Be specific — generic statements are not useful.
- problem_type_recommendation: Choose the single best fit from the enum. If truly mixed, 
  use HYBRID and explain the mix in suggested_ai_architecture.
- agentic_workflow: Only include steps that are genuinely AI-driven. Do not pad with 
  obvious infrastructure steps.
- technical_backlog: Tickets should be actionable stories, not vague epics. Include 
  at least one P0 and one P1 ticket.
- executive_summary: Write for a CTO who will share this with their board. 
  No jargon, concrete business outcomes, quantified where possible.
```

### Why this structure works
- Role framing improves output consistency across diverse inputs.
- "Return ONLY valid JSON" dramatically reduces prose leakage that breaks the parser.
- Schema injection in the prompt acts as a second-pass reminder after the role framing.
- Field guidance prevents the LLM from producing generic or padded content.

## 4. User Prompt Design

The user prompt is built per-request in `prompt_templates.build_user_prompt()`.

```
BUSINESS PROBLEM:
{problem_description}

{optional context block below}
ADDITIONAL CONTEXT:
{context}
```

### Design decisions
- The problem is presented verbatim — no preprocessing or summarization. The LLM should see the messiness and translate it.
- The context block is conditionally included only when provided, to avoid injecting empty or `null` strings that confuse the model.
- No few-shot examples in the user prompt (kept in system prompt via field guidance instead) to save tokens per call.

## 5. Output Parsing Strategy

```python
raw = await provider.complete(system_prompt, user_prompt)

try:
    parsed_dict = json.loads(raw)
    output = BlueprintOutput.model_validate(parsed_dict)
except json.JSONDecodeError as e:
    raise BlueprintParseError("Invalid JSON from LLM", raw_output=raw) from e
except ValidationError as e:
    raise BlueprintParseError("Schema validation failed", raw_output=raw) from e
```

**Fallback behavior:** On any parse error, the API returns HTTP 502 with:
- `code: "PARSE_ERROR"`
- `message: "LLM response could not be parsed into the expected schema"`
- `raw_llm_output`: the first 2000 characters of the raw response (for debugging)

This is preferable to returning partial or invalid data to the UI.

**No auto-retry for MVP.** Retrying on parse failure can produce the same bad output, wastes tokens, and adds latency. The v2 approach uses structured outputs (Anthropic's tool use / JSON mode with schema enforcement) which eliminates most parse failures.

## 6. Prompt Caching (Anthropic API)

When using the Anthropic provider, the system prompt is marked for caching:

```python
messages.create(
    model=self.model_id,
    system=[
        {
            "type": "text",
            "text": system_prompt,
            "cache_control": {"type": "ephemeral"}  # 5-minute TTL cache
        }
    ],
    messages=[{"role": "user", "content": user_prompt}],
    max_tokens=4096,
)
```

**Why this matters for the demo:** The system prompt is ~1500 tokens. On repeated calls (e.g., demoing 3 problems back-to-back), caching eliminates re-processing the system prompt, reducing latency by ~20–40% and cutting input token cost by ~90% for the cached portion.

## 8. Switching Providers

### Use mock provider (default, for tests and offline development)
```bash
LLM_PROVIDER=mock uvicorn app.main:app --reload
```

### Use Anthropic provider (for real blueprints)
```bash
LLM_PROVIDER=anthropic \
ANTHROPIC_API_KEY=sk-ant-... \
MODEL_ID=claude-sonnet-4-6 \
uvicorn app.main:app --reload
```

### Using a `.env` file
```env
LLM_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
MODEL_ID=claude-sonnet-4-6
```

The `MockLLMProvider` ignores all environment variables. The `AnthropicLLMProvider` will raise a `ConfigurationError` at startup if `ANTHROPIC_API_KEY` is empty.

## 9. Mock Provider Fixture Design

The mock response is a single JSON file at `tests/fixtures/mock_blueprint.json`. It uses a fictional logistics company (same as Canonical Problem A) as its content domain.

**Design principles for the fixture:**
- All 10 sections populated with realistic, detailed content (not placeholder text like `"TODO"` or `"..."`)
- `problem_type_recommendation` is `"HYBRID"` (exercises the most code paths)
- `technical_backlog` contains 8 tickets across P0–P2 and all phases
- `agentic_workflow` has 4 steps with distinct agent names and tool lists
- `risks_and_assumptions` has 5 items across all 4 risk categories
- Content is cohesive — all sections tell the same story about the same problem

This makes the Streamlit UI look fully populated and realistic during offline demos.

## 10. Quality Guardrails (v2 Roadmap)

These improvements are out of scope for MVP but address known failure modes:

| Failure Mode | v2 Solution |
|---|---|
| LLM returns prose instead of JSON | Anthropic structured outputs / tool use with schema |
| Empty or one-word section content | Section-level length validation + retry |
| Problem type misclassification | Multi-label classification sub-call before main generation |
| Hallucinated technology recommendations | RAG over a curated tech pattern library |
| Generic backlog tickets | Few-shot examples of good vs. bad tickets in system prompt |
