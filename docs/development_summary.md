## Purpose

A FastAPI + Streamlit application that accepts a business problem description and returns a structured, 14-section AI implementation blueprint — covering architecture, backlog, risk, testing, and an executive summary — generated by a large language model and validated before delivery.

Built as a MVP demonstrating AI Implementation Learning/Development/Leading skills.

## Spec-Driven Development

The project is built contract-first. Every component is constrained by a specification before any code is written.

**Living specification chain:**

```
CLAUDE.md  (engineering contract: standards, principles, what not to do)
    │
    ├── docs/PRD.md            (product requirements)
    ├── docs/ARCHITECTURE.md   (system design and provider strategy)
    ├── docs/API_SPEC.md       (endpoint contracts: paths, request/response shapes)
    └── docs/SPEC.md           (Pydantic schema reference)
```

`CLAUDE.md` acts as the AI pair-programmer's constitution — it defines coding standards, naming conventions, architecture principles, and explicit prohibitions. All changes must conform to it before being committed.

**Schema-first validation.** `app/models/schemas.py` is the single source of truth for every data shape in the system. All LLM output must pass `BlueprintOutput.model_validate()` before it touches the API response. Unvalidated output never reaches the caller.

**Prompt/schema alignment.** The system prompt in `app/core/prompt_templates.py` embeds the output schema explicitly. A dedicated test — `TestPromptTemplate.test_system_prompt_contains_all_required_output_fields` — compares the prompt's field references against the live Pydantic model. Any drift between the prompt and schema fails the test suite automatically.

**Documentation co-commits.** Docs are updated in the same commit as the behaviour change they describe — schema rename, endpoint change, and env var addition are never merged without a matching doc update.

## Git Versioning

All source history lives in a single remote on GitHub. Branch and commit discipline is enforced by convention and pre-commit hooks.

**Branch naming:**

| Prefix     | Use                         |
| ---------- | --------------------------- |
| `feat/`  | New functionality           |
| `fix/`   | Bug corrections             |
| `docs/`  | Documentation-only changes  |
| `chore/` | Dependency, config, tooling |

**Commit rules:**

- Imperative mood, present tense, ≤ 72 characters
- One logical change per commit — features and refactors are never bundled
- `--no-verify` is never used; hook failures are fixed, not bypassed
- No force-push to `main`; all merges via pull request

## CI/CD Pipeline

Every push and pull request runs the full pipeline on GitHub Actions (`.github/workflows/ci.yml`).

```
lint (ruff)
    │
type-check (mypy)
    │
    └── test (pytest + coverage)
              │
              └── docker-build  ← main branch only
```

**lint job** — `ruff check app/ tests/`
Catches style violations and import errors before any test runs.

**type-check job** — `mypy app/ --ignore-missing-imports`
Validates all function signatures and Pydantic field types statically.

**test job** — `pytest tests/ -v --cov=app --cov-report=xml --cov-fail-under=80`
Runs the full suite against `MockLLMProvider`. The `--cov-fail-under=80` flag is a hard gate — the job fails and blocks merge if coverage of `app/` falls below 80%.

**docker-build job** (main only) — `docker build -t ai-copilot-api .`
Confirms the production image builds cleanly after every merge.

**Render.com auto-deploy.** Both cloud services (`nexton-api`, `nexton-ui`) redeploy automatically on every push to `main` via the GitHub integration — no manual deployment steps after initial setup.

## Test Benchmarks and Approval Gates

The test suite is designed to be deterministic, fast, and free of external dependencies. All 40 tests pass with no network access and no API key set.

**Suite layout** (mirrors source layout):

| Test module                           | Covers                                                       |
| ------------------------------------- | ------------------------------------------------------------ |
| `tests/test_api.py`                 | FastAPI route handlers; request/response contracts           |
| `tests/test_blueprint_generator.py` | `BlueprintGenerator` orchestration; all 13 output sections |
| `tests/test_schemas.py`             | Pydantic model validation; prompt/schema alignment           |
| `tests/conftest.py`                 | Shared fixtures: test client, mock provider, sample requests |

**Key design rules:**

- `MockLLMProvider` returns a deterministic fixture (`tests/fixtures/mock_blueprint.json`) — no randomness, no timing sensitivity
- External dependencies (Anthropic SDK, OpenAI SDK) are never called from tests
- Internal logic is tested with real objects — `BlueprintGenerator` is never mocked in API tests
- Any test that could flake is a bug, not an acceptable risk

**Approval gates summary:**

| Gate           | Threshold          | Enforcement                            |
| -------------- | ------------------ | -------------------------------------- |
| Lint           | Zero violations    | CI blocks merge                        |
| Type check     | Zero errors        | CI blocks merge                        |
| Test pass rate | 100% (40/40)       | CI blocks merge                        |
| Coverage       | ≥ 80% of `app/` | CI hard fail (`--cov-fail-under=80`) |
| Docker build   | Clean build        | CI blocks merge on `main`            |

## Architecture Highlights

**Swappable LLM providers.** All LLM calls go through the `LLMProvider` abstract interface (`app/services/llm/base.py`). `MockLLMProvider`, `AnthropicLLMProvider`, and `OpenAILLMProvider` are interchangeable without touching route or generator code. Provider selection is resolved at request time via the `X-LLM-Provider` HTTP header.

**API key security.** Keys travel as HTTP request headers (`X-API-Key`) from the Streamlit sidebar to the FastAPI backend. They are used for that request only — never written to disk, never logged, never stored server-side, never returned in responses.

**Decoupled UI and API.** Streamlit calls FastAPI over HTTP. There are no shared imports between `ui/` and `app/`. The API is fully functional as a standalone service (curl, Postman, any HTTP client).

**No persistent storage.** Blueprints are ephemeral per request. No database, no file writes, no state between calls. The `.docx` export is generated in memory (`BytesIO`) and streamed directly to the browser.

**Single LLM call.** The entire 14-section blueprint is produced in one API call with a structured system prompt. Multi-step agentic pipelines are a documented future enhancement, not current scope.
